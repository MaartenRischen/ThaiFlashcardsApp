1. Set Creation Flow (Wizard UI)

The Set Wizard guides users through configuring their desired flashcard set.
  1. Welcome: Introduction screen.
  2. Proficiency Selection: User chooses their current Thai level (e.g., Beginner, Intermediate, Advanced).
  3. Topic Selection: User selects ONE topic type:
    • Custom Goal (Text input, e.g., "Talk about holiday plans")
    • Pre-defined Scenario (Chosen from a list based on proficiency level, e.g., "Ordering Food & Drinks")
    • Weird Scenario (Chosen from a list of surreal options, e.g., "Convincing a black hole to be more optimistic")
  4. Tone Selection: User adjusts a slider (1-10) to define the learning style (1 = Textbook realism, 10 = Maximum chaos).
  5. Review: Summary screen displays:
    • Selected Proficiency Level
    • Selected Topic (Value and Type: Goal/Scenario/Weird)
    • Selected Tone Level (Descriptive Label, e.g., "Actually funny")
    • Card Count Selection (Dropdown: 5, 10, 15, or 20 cards)
  6. Generation: User confirms, triggering the backend API call. The UI shows a loading/progress state.

2. API Call: `/api/generate-set` (POST)

The frontend sends the user's preferences to this backend endpoint to initiate set generation.

Request Body Structure

{ "preferences": { "level": "Advanced", // ProficiencyLevelString "specificTopics": "Convincing a black hole to be more optimistic", // Single topic string "toneLevel": 4, // Number (1-10) "topicsToDiscuss": "Convincing a black hole to be more optimistic" // Same as specificTopics (for prompt consistency) }, "totalCount": 15, // Number (5, 10, 15, or 20) "isTestRequest": false // Optional, boolean }

Backend Processing Steps

  1. Authentication: Verifies user identity using Clerk `auth()`. Returns 401 if not authenticated.
  2. Input Validation: Checks if `preferences` and `totalCount` exist. Returns 400 if missing.
  3. Call `generateCustomSet` (app/lib/set-generator.ts): Passes `preferences` and `totalCount`. This is the core phrase generation logic (see Section 3).
  4. Handle Test Requests: If `isTestRequest` is true, skips image generation and database operations, returns `generationResult` directly.
  5. Prepare Metadata: Creates initial `SetMetaData` object using `generationResult.cleverTitle` (generated by LLM), `preferences`, etc.
  6. Call `generateImage` (app/lib/ideogram-service.ts): Generates the set cover image (see Section 4). Uploads to storage.
  7. Database Operations (app/lib/storage.ts):
    • `addSetMetaData`: Saves set metadata (including image URL) to the database.
    • `saveSetContent`: Saves the generated phrases (`generationResult.phrases`).
    • `saveSetProgress`: Saves initial empty progress data for the user and set.
  8. Response:
    • On Success (201 Created): Returns `{ newSetMetaData: SetMetaData, ...generationResult }`.
    • On Failure (500 Internal Server Error): Returns `{ error: string }`. Attempts to delete partially created set data if an error occurred after metadata creation.

3. Core Generation Logic (`generateCustomSet`)

Located in app/lib/set-generator.ts. This function orchestrates LLM calls to generate phrases based on user preferences.

3.1. Temperature Calculation

3.2. Prompt Construction

A detailed prompt is built for the LLM, incorporating:

  • User's proficiency `level`.
  • The single `specificTopics` / `topicsToDiscuss` string.
  • The desired `totalCount` of phrases.
  • Instructions for output format (JSON array of `Phrase` objects).
  • Rules about politeness (`krap`/`ka`), pronunciation, mnemonics, and example sentences.
  • A request for a `cleverTitle` related to the topic.
  • Instructions to avoid generating phrases already present (passes `existingPhrases` array).

3.3. Batch Processing & LLM Calls

Generation happens in batches to handle larger `totalCount` requests and manage different LLM capabilities.

  • Uses `getBatchSize(model)` to determine how many phrases to request per API call (dynamic based on model).
  • Calls `generateOpenRouterBatch` which iterates through a prioritized list of models (`TEXT_MODELS`) via OpenRouter.
  • Uses calculated `temperature`.
  • Includes retry logic with exponential backoff for API/network errors.
  • Collects results from successful batches.

3.4. Response Parsing & Validation

  • Parses the LLM's JSON response.
  • Calls `validateAndCleanPhrases` for each generated phrase:
    • Checks for required fields (`english`, `thai`, `pronunciation`).
    • Performs basic type checks.
    • Applies quality checks (e.g., length, non-empty).
    • Note: Currently, stricter semantic validation based on `toneLevel` (checking if phrases make logical sense for low tones) is conceptual and not fully implemented in the validation step itself. The prompt instructs the LLM, but validation doesn't enforce it rigorously post-generation.
  • Filters out invalid phrases.

3.5. Error Handling & Fallback

  • Aggregates errors encountered during batch processing (API errors, parsing errors, validation errors).
  • If generation yields fewer phrases than `totalCount` (due to errors or LLM limits), it attempts to pad the results with fallback phrases from app/data/phrases.ts.
  • Returns a `GenerationResult` object containing:
    • `phrases`: Array of validated `Phrase` objects.
    • `cleverTitle`: String title suggested by the LLM.
    • `llmBrand`, `llmModel`: Info about the primary model used.
    • `temperature`: The temperature value used for generation.
    • `aggregatedErrors`, `errorSummary`: Details about any errors encountered.

4. Image Generation Logic (`generateImage`)

Located in app/lib/ideogram-service.ts. Called by the API route after successful phrase generation.
  • Receives a single `prompt` string.
  • Constructs the prompt using the `cleverTitle` or `specificTopics` from the generation result.
    const imagePrompt = `Cartoon style illustration featuring a friendly donkey and a bridge, related to the topic: "${topicDescription}". ` + `Use vibrant colors that are friendly and engaging. ` + `Crucial: do not include any text, letters, or numbers in the image.`;
  • No Tone Influence: The image prompt is not affected by the selected `toneLevel`.
  • Strict No Text/Letters/Numbers Rule: The prompt explicitly forbids any text, letters, or numbers.
  • Calls the Ideogram API via `callIdeogramApi`.
  • Handles polling for job completion.
  • Returns the generated image URL upon success, or `null` on failure.
  • Includes a fallback mechanism using a simpler prompt (`Simple cartoon illustration of a friendly donkey and a bridge doing something completely random. No text, letters, or numbers.`) if the primary generation or upload fails.
  • If both primary and fallback generation/upload fail, a static placeholder image is used.

5. Database & Storage Interaction (`storage.ts`)

Located in app/lib/storage.ts. Handles interaction with the database (Prisma/Supabase) and image storage (Supabase Storage).
  • Uses Prisma client for database operations (PostgreSQL).
  • `addSetMetaData`: Creates a new `FlashcardSet` record.
  • `saveSetContent`: Creates multiple `Phrase` records associated with the `FlashcardSet`.
  • `saveSetProgress`: Creates an initial `UserSetProgress` record linking the user and the new set.
  • `getAllSetMetaData`, `getSetContent`, `getSetProgress`, `updateSetProgress`, `deleteSetMetaData` handle fetching, updating, and deleting set data.
  • `uploadImageFromUrl` (in app/lib/imageStorage.ts): Fetches an image from a URL (e.g., Ideogram's result) and uploads it to Supabase Storage.
